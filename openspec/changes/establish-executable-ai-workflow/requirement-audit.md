# Legacy Requirement Audit

## Method

This audit compares each row of `docs/REQUIREMENT_LOG.md` with tracked source,
tests, and repository structure as of 2026-07-15. The legacy status and test
columns are claims to audit, not evidence. Statuses in this file mean:

- `implemented`: the tracked implementation satisfies the stated behavior.
- `partial`: a meaningful portion exists, but the stated behavior or its
  cross-surface parity is incomplete.
- `missing`: no implementation of the stated behavior was found in the
  relevant surface.
- `blocked`: the repository cannot currently verify or deliver the behavior
  because a prerequisite is unavailable.

An issue ID names an existing structured issue. `Superseded` means the audited
legacy wording or presentation was not retained as a product obligation; the
named current-state capability spec is authoritative instead.

## Migration Outcome

Implemented current behavior is captured in capability specs. Remaining product
obligations map to structured issue IDs and the evidence-based roadmap;
obsolete legacy presentation claims are explicitly superseded rather than
silently converted into backlog. The legacy log remains historical migration
input and is not a normative requirements source.
